# Agent Instructions — VibeSafe (바이브코딩 보안 스캐닝 서비스)

이 문서는 **WAT 프레임워크**(Workflows, Agents, Tools)에 따라 VibeSafe 프로젝트의 개발 에이전트가 따라야 할 지침이다.
확률적 AI(추론)와 결정론적 코드(실행)를 분리하여 시스템 신뢰성을 확보하는 것이 핵심 원칙이다.

---

## 1. 근본 문제 정의 (First Principles)

### 1.1 문제의 물리적 구조
바이브 코딩은 **개발 지식 없이 AI에 의존하여 실행 가능한 소프트웨어를 생성**하는 행위다.
전통적 개발 파이프라인에는 아래 보안 레이어가 존재한다:

| 레이어 | 전통 개발 | 바이브 코딩 |
|--------|-----------|------------|
| 코드 리뷰 (Peer Review) | 2인 이상 수동 검토 | 없음 |
| 정적 분석 (SAST) | CI/CD 파이프라인 내 SonarQube, Semgrep 등 | 없음 |
| 동적 분석 (DAST) | 스테이징 환경 OWASP ZAP 등 | 없음 |
| 의존성 감사 (SCA) | Snyk, Dependabot 자동화 | 없음 |
| 시크릿 스캐닝 | git-secrets, TruffleHog | 없음 |
| 보안 아키텍처 설계 | Threat Modeling (STRIDE/DREAD) | 없음 |

**결론: 바이브 코딩은 6개 보안 레이어가 전부 누락된 상태에서 프로덕션 코드를 배포한다.**

### 1.2 도메인별 위험 데이터 분류
바이브 코딩으로 만든 서비스의 데이터 유출 위험은 **서비스 도메인에 따라 노출 대상 데이터가 다르다:**

| 도메인 | 고위험 데이터 | 규제 | 주요 공격 벡터 |
|--------|-------------|------|--------------|
| 이커머스 | PCI DSS 카드번호, 결제 토큰, 배송 주소, 구매 이력 | PCI DSS, GDPR, 전자상거래법 | SQL Injection, XSS, IDOR |
| 게임 | 유저 인증 토큰, 인앱 결제 정보, 채팅 로그, 미성년자 데이터 | COPPA, GDPR | API Key 노출, WebSocket Hijacking |
| 플랫폼/SaaS | OAuth 토큰, 사용자 PII, API 키, 멀티테넌트 격리 | SOC2, GDPR, CCPA | BOLA, JWT 조작, SSRF |
| 헬스케어 | PHI(보호대상 건강정보), 진료 기록, 처방전 | HIPAA, 개인정보보호법 | 암호화 미적용, 접근 제어 우회 |
| 핀테크 | 계좌번호, 거래 내역, KYC 문서 | PCI DSS, 전자금융거래법 | 인증 우회, 중간자 공격 |
| 교육 | 학생 성적, 학부모 연락처, 학습 기록 | FERPA, COPPA | 권한 상승, 데이터 과다 노출 |

### 1.3 해결해야 할 구체적 문제
1. **보안 무지**: 바이브 코더는 OWASP Top 10이 무엇인지 모른다
2. **도구 부재**: 기존 SAST/DAST 도구는 CLI 기반으로 비개발자가 사용 불가
3. **도메인 무시**: 범용 스캐너는 PCI DSS vs HIPAA vs COPPA 같은 도메인 특화 규제를 구분하지 않음
4. **수정 불가**: 취약점을 발견해도 바이브 코더는 직접 수정할 기술적 역량이 없음

## 2. WAT 아키텍처 구조

### Layer 1: Workflows (지침서)
`workflows/` 디렉토리의 마크다운 SOP 파일들.
각 워크플로우는 **목표, 필수 입력값, 사용 도구, 예상 출력, 예외 처리 방법**을 정의한다.

### Layer 2: Agents (의사결정자)
Claude Code 또는 Antigravity 에이전트가 이 역할을 수행한다.
워크플로우를 읽고 → 도구를 순서대로 실행하고 → 실패 시 복구하고 → 필요 시 명확화 질문을 한다.

### Layer 3: Tools (실행기)
`tools/` 디렉토리의 Python/Node.js 스크립트.
API 호출, 파일 변환, 보안 스캐닝, DB 쿼리 등 결정론적 작업만 담당한다.

**정확도 근거:** AI가 5단계를 직접 처리하면 0.9^5 = 59% 성공률.
결정론적 스크립트에 실행을 위임하면 각 단계 99%+ → 전체 95%+ 성공률 유지.

## 3. 에이전트 운영 규칙

### 3.1 기존 도구 우선 확인
새 스크립트를 작성하기 전에 반드시 `tools/` 디렉토리를 확인한다.
워크플로우가 요구하는 작업에 해당하는 도구가 이미 존재하면 그것을 사용한다.

### 3.2 실패 시 학습-적응 프로세스
오류 발생 시:
1. 전체 에러 메시지와 스택 트레이스를 읽는다
2. 스크립트를 수정하고 재테스트한다 (유료 API 호출 시 사전 확인 필수)
3. 발견한 내용을 `workflows/error_log.md`에 기록한다
4. 해당 워크플로우를 업데이트한다

### 3.3 워크플로우 최신 유지
더 나은 방법을 발견하거나, 제약 조건을 파악하거나, 반복되는 이슈가 있으면 워크플로우를 업데이트한다.
**단, 명시적 지시 없이 워크플로우를 생성하거나 덮어쓰지 않는다.**

## 4. 자기 개선 루프 (Self-Improvement Loop)

모든 실패는 시스템을 강화하는 기회다:

```
[오류 발생] → [원인 식별] → [도구 수정] → [수정 검증] → [워크플로우 업데이트] → [다음 실행에 반영]
```

### 4.1 오류 분류 체계
| 등급 | 정의 | 대응 |
|------|------|------|
| P0-CRITICAL | 보안 스캔 결과 오탐/미탐, 데이터 유실 | 즉시 중단, 근본 원인 분석, 워크플로우 + 도구 동시 수정 |
| P1-HIGH | API 타임아웃, 파서 실패, 의존성 충돌 | 재시도 로직 추가, 폴백 경로 설정 |
| P2-MEDIUM | UI 렌더링 오류, 비효율적 쿼리 | 다음 스프린트에 수정 |
| P3-LOW | 로그 포맷 불일치, 코드 스타일 | 배치 수정 |

### 4.2 Feedback Loop 설계
```
입력: {실행파일, 도메인유형, 사용기술스택}
    ↓
[정적분석 도구 실행] → 결과 → [도메인별 규칙 엔진 매칭]
    ↓                              ↓
[동적분석 도구 실행] → 결과 → [위험도 점수 산출]
    ↓                              ↓
[자동 수정 제안 생성] ← [이전 스캔 결과 DB 참조]
    ↓
[사용자 피드백 수집: 오탐 신고, 수정 적용 여부]
    ↓
[규칙 엔진 가중치 조정] → [다음 스캔에 반영]
```

## 5. 디렉토리 구조

```
project/
├── .tmp/                    # 임시 파일 (스캔 중간 결과, 파싱 데이터). 재생성 가능.
├── .env                     # API 키, 환경 변수 (다른 곳에 절대 시크릿 저장 금지)
├── tools/                   # 결정론적 실행 스크립트
│   ├── scanner/
│   │   ├── sast_runner.py           # Semgrep 기반 정적 분석 실행기
│   │   ├── dast_runner.py           # OWASP ZAP 기반 동적 분석 실행기
│   │   ├── sca_runner.py            # 의존성 취약점 스캐너 (npm audit, pip-audit)
│   │   ├── secret_scanner.py        # 하드코딩된 API 키/비밀번호 탐지
│   │   └── domain_rule_engine.py    # 도메인별 규제 매칭 엔진
│   ├── remediation/
│   │   ├── auto_fix_generator.py    # AI 기반 수정 코드 제안
│   │   ├── patch_applier.py         # 자동 패치 적용기
│   │   └── fix_validator.py         # 수정 후 재스캔 검증
│   ├── report/
│   │   ├── pdf_generator.py         # 보안 리포트 PDF 생성
│   │   ├── score_calculator.py      # 보안 점수 산출 (0-100)
│   │   └── compliance_checker.py    # 규제 준수 여부 판정
│   └── infra/
│       ├── sandbox_manager.py       # Docker 샌드박스 생성/폐기
│       ├── file_extractor.py        # 업로드 파일 압축 해제 및 구조 분석
│       └── db_client.py             # PostgreSQL/Redis 클라이언트
├── workflows/               # 마크다운 SOP
│   ├── scan_submission.md           # 사용자 제출 → 스캔 완료까지 전체 흐름
│   ├── domain_classification.md     # 도메인 자동 분류 로직
│   ├── vulnerability_triage.md      # 취약점 분류 및 우선순위 결정
│   ├── auto_remediation.md          # 자동 수정 제안 생성 흐름
│   ├── report_generation.md         # 보안 리포트 생성 흐름
│   ├── error_log.md                 # 누적 오류 기록 및 학습 내용
│   └── deployment.md                # 배포 프로세스
├── credentials.json          # OAuth 인증 (gitignored)
├── token.json                # 토큰 저장 (gitignored)
└── README.md
```

### 파일 저장 원칙
- **최종 산출물**: 클라우드 서비스(S3, GCS)에 저장하여 사용자가 직접 접근
- **중간 파일**: `.tmp/`에 저장, 언제든 재생성 가능
- **로컬 파일은 처리용**. 사용자가 봐야 할 것은 클라우드에 저장

## 6. 제약 조건

### 6.1 절대 금지 사항
- `.env` 외 위치에 API 키, 비밀번호, 토큰 저장 금지
- 사용자 업로드 파일을 샌드박스 밖에서 실행 금지
- 스캔 결과를 암호화 없이 외부 전송 금지
- 사용자 소스코드를 학습 데이터로 수집 금지

### 6.2 성능 제약
- 단일 스캔 타임아웃: 300초 (초과 시 부분 결과 반환 + 알림)
- 최대 업로드 파일 크기: 500MB
- 동시 스캔 큐: 사용자당 3건
- 도메인 규칙 엔진 응답 시간: 200ms 이내

## 핵심 원칙

> 너는 사용자의 의도(워크플로우)와 실제 실행(도구) 사이에 위치한다.
> 지침을 읽고, 판단하고, 올바른 도구를 호출하고, 오류에서 복구하고, 시스템을 지속적으로 개선한다.
> 실용적으로. 신뢰할 수 있게. 계속 학습하면서.
